# CODEOWNERS First-Run Setup

This file is loaded when `.github/CODEOWNERS` does not exist in the target repo.
Follow these steps exactly before proceeding to any code analysis.

---

## Why This Matters

CODEOWNERS is the enforcement layer for P0 override approvals. Without it,
the skill cannot tag the right people or enforce the 2-approver rule on team
projects. First-run setup takes about 2 minutes and only happens once per repo.

---

## Step 1 — Identify Existing Contributors

Run the following command to extract contributors from git history:

```bash
git log --format="%ae %an" | sort -u
```

Present the results to the user:

> "I found the following contributors in this repo's git history. I'll use this
> to suggest your team list. You can add anyone not listed here by GitHub username.
>
> [list names and emails from git log output]"

---

## Step 2 — Collect Developer and Approver Lists

Ask the user:

> "Please confirm or update the developer list, and tell me which GitHub usernames
> should be designated as **approvers**.
>
> Approvers are the people required to sign off on any P0/Critical override in a
> team project. You need at least one approver — two or more is strongly recommended.
>
> Format your response as:
> - Developers: @username1, @username2, ...
> - Approvers: @username1, @username2, ..."

Wait for the user's response. Record:
- `developers`: list of GitHub usernames (without `@`)
- `approvers`: list of GitHub usernames (without `@`)

If the user provides only one approver, note:
> "One approver recorded. Note that P0 overrides require 2+ approvals — with only
> one approver, all P0 blocks will be unoverridable unless you add a second approver
> or enable solo-dev mode."

---

## Step 3 — Determine Solo or Team Mode

Ask the user:

> "Is this a solo project (just you) or a team project?
>
> - **Solo:** You can override P0 blocks yourself, but every override is recorded
>   in the findings report with a timestamp and your stated reason.
> - **Team:** P0 blocks require 2+ approver sign-offs on the PR before merge.
>
> Which applies?"

Record `solo_dev: true` or `solo_dev: false` based on the response.

---

## Step 4 — Determine Target Branch

Ask the user:

> "What is your primary integration branch — the one PRs merge into?
> (Common values: `main`, `master`, `develop`)"

Record as `target_branch`.

---

## Step 5 — Generate .github/CODEOWNERS

Create the directory if it doesn't exist:

```bash
mkdir -p .github
```

Write `.github/CODEOWNERS` using the values collected above:

```
# NASA/DOD Code Review — Auto-generated by nasa-dod-code-review skill
# Edit this file to update ownership and approver assignments.
#
# Developers (informational — tracked but not enforced by GitHub reviews)
# @dev1 @dev2 @dev3
#
# Approvers — required to approve P0/Critical overrides on this repo
* @approver1 @approver2
#
# Language-specific ownership (uncomment and customize as needed)
# *.c        @approver1
# *.py       @approver2
# *.go       @approver1 @approver2
# *.rs       @approver1
# *.java     @approver2
# *.ts *.js  @approver1 @approver2
```

Replace `@dev1`, `@dev2`, `@approver1`, `@approver2` with the actual GitHub
usernames collected in Steps 2 and 3.

---

## Step 6 — Generate .github/nasa-dod-review-config.json

Write `.github/nasa-dod-review-config.json` with the collected values:

```json
{
  "solo_dev": false,
  "developers": ["dev1", "dev2"],
  "approvers": ["approver1", "approver2"],
  "last_full_scan": null,
  "target_branch": "main",
  "full_scan_max_age_days": 7
}
```

Replace values with what was collected:
- `solo_dev`: `true` or `false`
- `developers`: array of GitHub usernames without `@`
- `approvers`: array of GitHub usernames without `@`
- `last_full_scan`: always `null` on first run
- `target_branch`: the branch name collected in Step 4
- `full_scan_max_age_days`: default `7` — user may change this to any positive integer

---

## Step 7 — Commit

Before committing, confirm with the user:

> "Ready to commit `.github/CODEOWNERS` and `.github/nasa-dod-review-config.json`
> to this repository. Proceed?"

Wait for confirmation. Then:

```bash
git add .github/CODEOWNERS .github/nasa-dod-review-config.json
git commit -m "chore: add CODEOWNERS and nasa-dod review config"
```

---

## Step 8 — Confirm and Continue

Tell the user:

> "First-run setup complete.
>
> - `.github/CODEOWNERS` created with [N] approver(s): [list them]
> - `.github/nasa-dod-review-config.json` created ([solo/team] mode, target branch: [branch])
>
> Continuing with the code review now."

Return to `README.md` Step 2 and continue the review.

---

## Editing Later

To update the approver list or switch between solo/team mode after first run:
1. Edit `.github/CODEOWNERS` to change the `*` line
2. Edit `.github/nasa-dod-review-config.json` to update `approvers` and/or `solo_dev`
3. Commit both files

There is no need to re-run this template for configuration changes.
