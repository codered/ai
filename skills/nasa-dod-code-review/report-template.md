# Report Template

Use this template to format the final code review report. Generate it twice:
1. Output the full report to terminal/chat
2. Write it to `reports/code-review-YYYY-MM-DD-HHMMSS.md` in the target repo
   (create `reports/` directory if it doesn't exist; include time to avoid same-day collisions)

The findings come from your analysis passes in `reviewer-prompt.md`. Format
each raw finding using the **Finding Entry** structure below. Every finding
gets three fix options with language-correct code.

## Tone Reminders by Severity

**P0/Critical:** Assertive, no hedging. The word "must" not "should."
> "🔴 P0/Critical — This blocks merge. [failure mode]. This is non-negotiable in safety-critical code."

**P1/Major:** Direct and clear. Explain the risk plainly.
> "🟠 P1/Major — This needs to be resolved before the next release. [explanation]."

**P2/Minor:** Constructive and helpful.
> "🟡 P2/Minor — Easy to clean up. [what and why]. Three options:"

**P3/Advisory:** Light, optional framing.
> "🔵 P3/Advisory — No action required, but here are some options if you'd like to refine this:"

**Override note (P0, solo dev):** Factual, no judgment.
> "Override recorded. Developer acknowledged risk. Proceeding."

**Override note (P0, team):**
> "Override requires 2 or more peer approvals on this PR. Approvers have been tagged above."

## Worked Example — P0 Finding Entry

```markdown
### [P0/Critical] Unbounded loop with no termination guarantee

**File:** `src/controllers/flight.c:142`
**Rule:** NASA Power of Ten — Rule 2 (bound all loops)
**Language:** C

**Why this must be fixed:** An unbounded loop in a flight-critical control loop
can cause the processor to hang indefinitely with no recovery path. On an embedded
target with a hardware watchdog, this triggers a system reset mid-flight. Without a
watchdog, the system deadlocks silently.

#### Fix Options

**Option 1 — Fixed iteration cap** *(recommended)*
Add a compile-time constant as the loop upper bound — the maximum number of iterations
is known at design time and can be enforced statically.
- ✅ Zero runtime overhead; auditable; satisfies NASA Power of Ten Rule 2 directly
- ✅ Detectable by static analysis tools
- ❌ Requires knowing the maximum iteration count at design time

```c
#define MAX_FLIGHT_LOOP_ITERATIONS 1000U

for (uint32_t i = 0U; i < MAX_FLIGHT_LOOP_ITERATIONS; i++) {
    if (target_reached()) { break; }
    update_flight_state();
}
/* Log if loop exhausted without reaching target */
```

**Option 2 — Watchdog-guarded loop**
Bound the loop by elapsed time using a monotonic timer, exiting gracefully if
the deadline is exceeded.
- ✅ Correct when the iteration count is genuinely unknown at design time
- ❌ Adds timing dependency; behavior varies under load; harder to test deterministically

```c
const uint32_t DEADLINE_MS = 50U;
uint32_t start = monotonic_time_ms();

while (!target_reached()) {
    if ((monotonic_time_ms() - start) >= DEADLINE_MS) {
        log_error("flight_loop: deadline exceeded");
        handle_timeout();
        break;
    }
    update_flight_state();
}
```

**Option 3 — Recursive approach with explicit depth limit**
Restructure the loop as tail recursion with a depth counter, making termination
formally provable by inspection.
- ✅ Termination is trivially provable — depth counter is decremented on every call
- ❌ Stack depth grows with iteration count; may exceed stack limits on embedded targets
- ❌ Recursion without a depth guard is prohibited by the NASA/JPL Institutional Coding Standard

```c
#define MAX_RECURSION_DEPTH 1000

void flight_loop(uint32_t depth) {
    if (depth == 0U || target_reached()) { return; }
    update_flight_state();
    flight_loop(depth - 1U);
}
/* Call as: flight_loop(MAX_RECURSION_DEPTH); */
```

**Approvers notified:** @jane-doe @mark-smith
```
